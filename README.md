# Тестовое задание

## Структура проекта

- bot - реализация Telegram-бота
- service - вспомогательные модули для работы программы
    - calculate - функции для расчетов
    - csv_writer - функции для записи в csv-файл
- api.py - модуль для запросов к API
- main.py - точка входа в программу
- models.py - модуль для описания моделей данных
- settings.py - модуль для чтения переменных окружения

## Используемые компоненты

- Aiogram - реализация бота
- APScheduler - библиотека для планирования выполнения кода

## Требования к переменным окружения

- ACCESS_TOKEN - токен доступа к APi
- BOT_TOKEN - токен доступа к Telegram-боту
- CHAT_ID - идентификатор Telegram-чата в который идет рассылка

## Выбор хранилища вместо google sheets

Так как данные включают в себя такие процессы как загрузка, обработка
хранение (ETL) и учитывая что данных (по ТЗ) около 100000, для перехода
с google sheets на мой взгляд хорошо впишется БД DuckDB. НО!!!
данная библиотека поддерживает только локальную работу на 1 клиенте.
Поэтому если планируется работа с данными на нескольких машинах
этот вариант отпадает и выбор падает на ClickHouse.

Причины:
 - DuckDB использует колоночную структуру что
хорошо подходит для аналитки данных в реальном времени
 - минимизирует объем передаваемых данных благодаря колоночному
хранению и сжатию данных
 - есть расширения для интеграции с ИИ
 - интегрируется прямо в python (по типу SQLite)

Другие варианты:
- PostgreSQL - неподходит для больших объемов данных: 
более высокая стоимость хранения (почти в 10 раз больше при 
100 млн строк), меньшая скорость работы при больших
обьъемах данных (почти в 2 раза больше DuckDB, для аналитики такое себе)
- ClickHouse - тоже отличный вариант для аналитики данных, 
скорее даже более предпочтителен чем DuckDB т.к отлично работает с 
гораздо более большими объемами данных, но учитывая что 
по легенде данные до этого находились в google sheets, решил не 
перепрыгивать через 2 ступеньки.
